最开始做博客时，我也认真考虑过后台系统。

它们看起来都很方便：有可视化编辑器、有媒体管理、有分类和标签面板，还有各种插件。问题在于，当博客真的进入长期维护阶段，你会慢慢发现自己真正依赖的不是这些功能，而是内容本身是不是足够自由。

## Markdown 给我的第一个好处，是简单

Markdown 没什么门槛，几分钟就能重新捡起来。

它就是普通文本，不依赖某个特定平台，也不绑定某个后台数据结构。无论我用 VS Code、手机记事本还是临时开的在线编辑器，都能继续写下去。

而且 Markdown 和 Git 的配合非常自然。

每次修改文章时，我都能知道：

- 哪一段被删掉了
- 哪一句被改写了
- 哪个标题结构被重做了

这种“内容有历史”的感觉，会让写作变得很踏实。

## 后台并不是没价值

我并不觉得后台不好。

它真正擅长解决的是这些问题：

- 团队协作
- 多人审核
- 可视化编辑
- 非技术成员参与发布

如果是团队站点，这些能力非常重要。但如果博客的维护者只有我自己，那我更在意的其实是“写完之后能不能立刻发出去”。

## 个人博客更需要的是低摩擦

我越来越相信，一个人能不能稳定更新博客，往往取决于发布动作是不是足够轻。

如果每次写完都还要切到另一个后台、再处理一串表单、再检查格式、再补媒体信息，写作时刚积累起来的那点状态就很容易散掉。

相反，如果流程只是：

1. 新建一个 Markdown 文件
2. 写完正文和 frontmatter
3. 提交仓库
4. 自动部署

那整件事就会非常顺。你写的是内容，而不是在操作系统。

## 最后

我不是在反对后台，我只是越来越在意一个问题：

> 这套方案会不会让我三个月后还愿意继续写？

如果答案是否定的，那它功能再完整，对我来说也不是合适的博客方案。
