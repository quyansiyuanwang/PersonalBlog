这不是我第一次想认真做一个博客。

过去几年里，我试过 WordPress、静态站主题、文档站改造，甚至还想过给自己搭一个小型 CMS。每次开始时都很兴奋，但真正停更的原因几乎都一样：系统越来越完整，写作反而越来越靠后。

所以这一次，我不再从“功能列表”开始，而是先给自己定了几个更朴素的原则。

## 先决定内容放在哪里

我希望文章和代码一样，都能被版本管理。

这件事带来的好处非常直接：

- 改了哪一句，一眼就能看到
- 旧版本可以回退，不怕误删
- 未来如果换技术栈，内容也不会丢

对个人博客来说，内容的归属感比后台是否华丽更重要。

## 再决定维护动作要足够轻

我很了解自己。如果每发一篇文章都要经历上传、排版、切后台、检查字段这些步骤，我更新频率一定会越来越低。

所以这次我希望发布流程尽量短：

1. 在仓库里新建 Markdown 文件
2. 写完 frontmatter 和正文
3. 推送到 GitHub
4. 让 GitHub Pages 自动部署

当步骤足够少时，写作才不会被部署动作打断。

## 最后才是页面应该长什么样

我喜欢有一点气质的界面，但不喜欢太吵的界面。

博客不是作品集首页，也不是产品发布页。它最核心的任务不是展示“我会做很多动效”，而是让人愿意把一篇文章读完。

所以我给这个站点定的视觉方向很明确：

- 阅读优先
- 轻装饰
- 中性色与纸张感背景
- 尽量让页面退后一步

## 技术选型服务于长期写作

我最后选了 `Vue 3 + Vite + Markdown + GitHub Pages` 这条路线，不是为了追新，而是因为它刚好平衡了几件事：

- 足够现代
- 足够稳定
- 足够容易维护
- 足够方便继续扩展

以后如果要补 RSS、搜索、评论或者 SEO，也都有继续扩展的空间。

## 写在最后

现在回头看，我觉得重新开始做博客最重要的不是“选对工具”，而是先回答两个问题：

1. 你想长期记录什么？
2. 你愿意用多复杂的方式维护它？

很多时候，答案越简单，事情反而越容易坚持下去。
