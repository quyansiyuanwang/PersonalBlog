如果博客只是偶尔更新，一套完整后台似乎很方便。

但只要你开始认真累积内容，就会慢慢意识到：**内容归属权** 和 **可迁移性** 才是更重要的事。

## 我为什么更偏向 Markdown

首先，Markdown 几乎没有学习成本。

它足够轻，也足够稳定。你不用担心某个后台停止维护，也不用担心导出格式变得麻烦。文章就是普通文本文件，任何编辑器都能打开，任何仓库都能保存。

其次，Markdown 和版本管理天然契合。

每次改动都可以被记录：

- 哪一段被删掉了
- 哪一句被改写了
- 哪个标题结构被重做了

这种清晰感，会让长期写作变得非常踏实。

## 后台系统真正解决了什么

后台当然不是没有价值。

它解决的是：

- 团队协作
- 多人审核
- 可视化编辑
- 非技术成员参与发布

但如果博客的作者只有你自己，那么这些能力很可能不是最优先的。

## 个人博客更需要的是低摩擦

我越来越相信，个人博客的更新频率，往往取决于发布动作是不是足够轻。

如果写完一篇文章以后，还要打开另一个系统、调整样式、确认字段、处理图片、修正格式，热情很容易在这些动作里被消耗掉。

相反，如果流程只是：

1. 新建一个 Markdown 文件
2. 写完正文和 frontmatter
3. 提交仓库
4. 自动部署

那么写作本身就会变得更连贯。

## 最后

我不是反对后台，而是更在意一个问题：

> 这套方案会不会让我三个月后还愿意继续写？

如果答案是否定的，那它再完整，也不是适合我的博客系统。
