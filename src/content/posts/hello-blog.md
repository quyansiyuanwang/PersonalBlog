每次想重新开始写博客时，我都会先提醒自己一件事：**不要一上来就追求“完整”**。

过去我也试过很多方案。功能最全的后台、模板很多的主题、可视化很强的 CMS，都很吸引人。但真正坚持不下去的原因，往往不是功能太少，而是维护成本太高。

所以这一次，我先做了四个决定。

## 1. 内容必须和代码一样，放进版本管理

我希望文章本身是稳定的、可回滚的、能追踪修改历史的。

把内容放进仓库，最直接的好处有三个：

1. 每次修改都能看到差异
2. 内容和站点结构不会分离
3. 迁移成本很低

对我来说，这比“后台编辑有多华丽”更重要。

## 2. 发布流程必须足够轻

如果每写一篇文章，都要额外经历上传、排版、配置、修样式，那么我很快就会停止更新。

所以我给自己定的规则是：

- 文章使用 Markdown
- 构建使用现成脚本
- 发布交给 GitHub Actions

这样一来，写作和部署之间只剩下非常短的一条路径。

## 3. 样式要有气质，但不能压过内容

我喜欢有质感的页面：温和的颜色、留白、清晰的标题层级、克制的装饰。

但博客不是作品集首页，也不是产品 landing page。真正需要被记住的不是过度设计的动画，而是内容本身。

所以这个站点的视觉方向定得很明确：

- 阅读优先
- 轻装饰
- 中性色与纸张感背景
- 尽量让页面退后一步

## 4. 技术选型服务于长期写作，而不是短期炫技

我选择 `Vue 3 + Vite + Element Plus + Markdown + GitHub Pages`，并不是因为这组方案“最先进”，而是因为它刚好落在一个舒服的平衡点上：

- 足够现代
- 足够稳定
- 足够容易维护
- 足够方便继续扩展

未来如果要加 RSS、搜索、SEO、暗色模式、封面图、评论系统，依然有空间。

## 最后

如果你也准备开始写自己的博客，我很建议先问自己两个问题：

1. 你想长期记录什么？
2. 你愿意用多复杂的方式维护它？

很多时候，答案越简单，越容易坚持下去。
